# Harness 0.1 执行模型

## 1. 文档定位

本文档只定义 `Harness 0.1` 的最小执行模型。

0.1 当前不讨论：

- task object model
- batch / execution unit
- 生命周期内核
- runtime control plane

本文档只回答一个问题：

**在 0.1 中，复杂需求如何从讨论进入执行。**

---

## 2. 0.1 执行模型的目标

0.1 的执行模型目标不是把所有执行状态正式建模，而是先建立最小执行闭环。

这个闭环至少要满足：

- 需求不能一上来就直接进入实现
- AI 要先确认是否理解需求
- 在执行前，用户要能看见边界和计划
- 执行阶段可以由主 agent 调度查询子 agent 与按需执行子 agent
- 最终要输出代码和文档结果

---

## 3. 输入类型

0.1 当前支持的输入可以是：

- 一句话需求
- PRD
- 多轮讨论结果
- Bug 描述
- 小范围实现目标

这些输入在进入系统时，都先被视为：

- 待理解需求
- 待澄清问题
- 待分析输入

也就是说，0.1 的第一步不是直接执行，而是先进入理解确认。

---

## 4. 第一步：复述需求

0.1 的第一个正式口令是：

- `复述需求`

它的职责不是生成方案，而是确认 AI 是否真正理解需求。

### 4.1 `复述需求` 必须展示的内容

建议固定展示：

- 目标理解
- 当前边界
- 关键约束
- 待确认点

### 4.2 `复述需求` 不负责什么

在这一阶段，不要求展示：

- 详细计划
- 详细步骤
- 完整验证方案

这些属于后续分析/规划阶段。

---

## 5. 第二步：分析与规划

`复述需求` 之后，系统进入分析/规划阶段。

0.1 当前不把这一步做成额外正式口令，而是默认由：

- `PIR` 的 `Planner`
- `superpowers` 的：
  - `brainstorming`
  - `writing-plans`

共同承接。

这一阶段的目标是产出：

- 收敛后的边界
- 初步计划
- 风险 / 待确认点

---

## 6. 第三步：开始执行

0.1 的第二个正式口令是：

- `开始执行`

它不是普通指令，而是最小执行闸门。

它的意义是：

- 在此之前，系统仍处于理解和规划状态
- 在此之后，系统进入真正的执行状态

---

## 7. 执行前确认

在 0.1 中，`开始执行` 之前必须看到一份执行前确认摘要。

这一步不是正式口令，而是系统自动展示的确认步骤。

### 7.1 执行前确认必须展示什么

建议固定展示：

- 目标
- 边界
- 计划
- 风险 / 待确认点

### 7.2 执行前确认的作用

这一层确认的作用是：

- 让用户在执行前看见关键内容
- 防止边界不清时直接进入实现
- 让 `开始执行` 真正成为执行闸门，而不是一句随口指令

---

## 8. 第四步：进入执行阶段

执行前确认成立后，系统进入执行阶段。

这一阶段由以下结构承接：

- 主 agent
- 查询子 agent
- 按需执行子 agent

其中：

- 主 agent 负责主流程推进
- 查询子 agent 负责查代码、查文档、查上下文
- 执行子 agent 按需参与复杂实现任务

  0.1 中，这一层只做最小协作，不做复杂运行时编排系统。

---

## 9. 输出结果

0.1 执行结束后，系统至少应输出：

- 代码结果
- 文档结果
- 必要的执行说明

当前不要求：

- 完整提交控制协议
- 自动 git 提交
- 复杂交付状态机

---

## 10. 0.1 一句话执行流

0.1 当前最小执行流为：

1. 用户输入需求
2. `复述需求`
3. 分析 / 规划
4. `开始执行`
5. 系统展示执行前确认
6. 主 agent 进入执行
7. 产出代码与文档结果

---

## 11. 一句话总结

**Harness 0.1 的执行模型不是完整执行内核，而是一条由 `复述需求`、分析/规划、`开始执行`、执行前确认与最小主/子 agent 协作组成的最小执行闭环。**
