# Harness 0.1 产品范围

## 1. 文档定位

本文档用于定义 `Harness 0.1` 的最小产品范围。

本文档重点回答的问题是：

- 0.1 到底要做什么
- 0.1 明确不做什么
- 0.1 为什么已经足够形成最小产品闭环
- 0.1 的用户接入结果应该是什么

---

## 2. 0.1 的产品目标

0.1 的目标不是做完整平台，也不是做复杂执行内核。

0.1 的目标是：

**做一个可以正常发布、正常安装、正常使用的最小 Codex-first 产品闭环。**

它至少应做到：

- 用户能把产品接入到项目里
- 接入后项目具备最小可用的 Codex 工作骨架
- 用户能按文档规范和 PIR 协作
- 用户能使用接好的 `superpowers` workflow
- 系统支持主 agent、查询子 agent、按需执行子 agent
- 系统支持最基本口令
- 用户能形成最小协作闭环

---

## 3. 0.1 要做什么

### 3.1 最小安装能力

必须提供正式接入入口，例如：

```bash
harness-codex init
```

### 3.2 最小配置入口

必须写入：

- `.codex/config.toml`

### 3.3 最小项目规则入口

必须写入：

- `AGENTS.md`

### 3.4 最小文档治理入口

必须写入：

- `documents/README.md`

并建立文档是正式产物的最小骨架。

### 3.5 Codex 工作手册

必须提供：

- `documents/codex-pir/`

至少包含：

- roles
- workflows
- configuration
- prompts
- 安装接入说明
- init 生成资产说明

### 3.6 接入 `superpowers`

必须做到：

- 将 `superpowers` 作为 workflow / skill 层接入
- 使其服务于 `PIR`
- 不把它提升成内核

### 3.7 最小 agent 协作结构

必须支持：

- 主 agent
- 查询子 agent
- 按需执行子 agent

但先不做复杂 runtime 编排系统。

### 3.8 基本口令

必须至少支持：

- `复述需求`
- `开始执行`

---

## 4. 0.1 不做什么

以下能力明确延后：

- 生命周期内核
- task object model
- batch / execution unit
- runtime control plane
- blocked / resume 协议
- diagnostics 闭环
- 多宿主支持
- graph execution engine
- 重型 `doctor / upgrade`
- skill marketplace
- 复杂模板系统
- 项目类型细粒度配置系统

一句话说：

**0.1 不做重内核，只做最小工作骨架。**

---

## 5. 0.1 用户需要输入什么

0.1 应尽量做到：

**用户侧零配置。**

也就是说，用户在接入前不需要先提供：

- 开发语言
- 项目类型
- 框架类型

系统应优先：

- 自动扫描项目目录
- 自动识别项目特征
- 自动写入最小骨架

只有在识别不准时，才再向用户发起最小确认。

---

## 6. 0.1 系统自动做什么

系统应自动完成：

- 扫描目录结构
- 识别常见项目特征
- 写 `.codex/config.toml`
- 写 `AGENTS.md`
- 建 `documents/README.md`
- 建 `documents/codex-pir/`
- 建最小文档入口
- 建 skill 入口约定
- 输出初始化结果

---

## 7. 0.1 最小写入资产

第一版最小写入资产应包括：

- `.codex/config.toml`
- `AGENTS.md`
- `documents/README.md`
- `documents/codex-pir/README.md`

建议一并写入：

- `documents/codex-pir/01-roles.md`
- `documents/codex-pir/02-workflows.md`
- `documents/codex-pir/03-configuration.md`
- `documents/codex-pir/04-prompts.md`
- `documents/codex-pir/05-official-mapping.md`
- `documents/codex-pir/06-installation-and-project-integration.md`
- `documents/codex-pir/07-init-generated-assets.md`

---

## 8. 0.1 最小闭环

0.1 最终应该能完成这条链：

1. 用户在项目里执行 `harness-codex init`
2. 项目被写入最小骨架
3. 用户按 `复述需求` 开始
4. 系统进入 `Planner`
5. 必要时调用查询子 agent
6. 按 `superpowers` workflow 形成计划
7. 用户触发 `开始执行`
8. 主 agent 按需调度执行子 agent
9. 输出代码与文档产物

这就已经构成最小闭环。

---

## 9. 0.1 一句话定义

**Harness 0.1 是一个可安装、可接入、可开始使用的最小 Codex-first 协作产品：用文档规范、PIR、superpowers、主/子 agent 最小协作与基本口令，完成复杂需求的最小稳定执行闭环。**
