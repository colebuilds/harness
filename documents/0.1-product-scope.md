# Harness 0.1 产品范围

## 1. 文档定位

本文档用于定义 `Harness 0.1` 的最小产品范围。

本文档重点回答的问题是：

- 0.1 到底要做什么
- 0.1 明确不做什么
- 0.1 为什么已经足够形成最小产品闭环
- 0.1 的用户接入结果应该是什么

---

## 2. 0.1 的产品目标

0.1 的目标不是做完整平台，也不是做复杂执行内核。

0.1 的目标是：

**做一个可以正常发布、正常安装、正常使用的最小 Codex-first 产品闭环。**

它至少应做到：

- 用户能把产品接入到项目里
- 接入后项目具备最小可用的 Codex 工作骨架
- 用户能按文档规范和 PIR 协作
- 用户能使用完整受管的 `superpowers` workflow
- 系统支持主 agent、查询 agent、按需执行 agent
- 系统支持最基本口令
- 系统支持最小项目策略配置与一致性检查
- 用户能形成最小协作闭环

---

## 3. 0.1 要做什么

### 3.1 最小安装能力

必须提供正式接入入口，例如：

```bash
harness-codex init
```

### 3.2 最小配置入口

必须写入：

- `.codex/config.toml`

### 3.3 最小项目规则入口

必须写入：

- `AGENTS.md`

### 3.4 最小文档治理入口

必须写入：

- `documents/README.md`

并建立文档是正式产物的最小骨架。

### 3.5 最小运行时契约入口

必须提供：

- `.harness/runtime-contract.json`

用于固定：

- `复述需求` / `开始执行` / review 请求的映射
- `PIR` 与 `plan / dev / review` profiles 的关系
- `superpowers` 与运行时 agent 架构的关系

### 3.6 接入 `superpowers`

必须做到：

- 将 `superpowers` 作为 workflow / skill 层接入
- 使其服务于 `PIR`
- 不把它提升成内核

### 3.7 最小 agent 协作结构

必须支持：

- 主 agent
- 查询 agent
- 按需执行 agent

但先不做复杂 runtime 编排系统。

### 3.8 最小校验入口

必须提供：

- `harness-codex verify`

### 3.9 基本口令

必须至少支持：

- `复述需求`
- `开始执行`

---

## 4. 0.1 不做什么

以下能力明确延后：

- 生命周期内核
- task object model
- batch / execution unit
- runtime control plane
- blocked / resume 协议
- diagnostics 闭环
- 多宿主支持
- graph execution engine
- 重型 `doctor / upgrade`
- skill marketplace
- 复杂模板系统
- 项目类型细粒度配置系统

一句话说：

**0.1 不做重内核，只做最小工作骨架。**

---

## 5. 0.1 用户需要输入什么

0.1 当前保留的最小初始化配置为：

- `developer_language`
- `document_mode`
- `debug_mode`

其余行为固定为：

- 完整受管接入 `superpowers`
- 启用 `复述需求`
- 启用 `开始执行`
- 统一备份后再替换冲突文件

---

## 6. 0.1 系统自动做什么

系统应自动完成：

- 扫描目录结构
- 写 `.codex/config.toml`
- 写 `.harness/project-policy.json`
- 写 `.harness/components.lock.json`
- 写 `.harness/runtime-contract.json`
- 写 `AGENTS.md`
- 建 `documents/README.md`
- 按模式建文档目录
- 建项目策略 skill
- 完整接入受管 `superpowers`
- 输出初始化结果

---

## 7. 0.1 最小写入资产

第一版最小写入资产应包括：

- `.codex/config.toml`
- `.harness/project-policy.json`
- `.harness/components.lock.json`
- `.harness/runtime-contract.json`
- `AGENTS.md`
- `documents/README.md`
- `skills/harness-project-policy/SKILL.md`
- `/.harness/superpowers/`

若 `document_mode=full`，建议一并写入：

- `documents/requirements/README.md`
- `documents/designs/README.md`
- `documents/deliveries/README.md`
- `documents/evolution/README.md`
- `documents/standards/ai-collaboration/README.md`

---

## 8. 0.1 最小闭环

0.1 最终应该能完成这条链：

1. 用户在项目里执行 `harness-codex init`
2. 项目被写入最小骨架
3. 项目获得 `project-policy`、`runtime-contract` 与完整受管 `superpowers`
4. 用户按 `复述需求` 开始
5. 系统进入 `Planner`
6. 必要时调用查询 agent
7. 按 `superpowers` workflow 形成计划
8. 用户触发 `开始执行`
9. 主 agent 按需调度执行 agent
10. 输出代码与文档产物
11. 用户可运行 `harness-codex verify`

这就已经构成最小闭环。

---

## 9. 0.1 一句话定义

**Harness 0.1 是一个可安装、可接入、可开始使用的最小 Codex-first 协作产品：用文档规范、PIR、受管 `superpowers`、主 / 查询 / 执行 agent 最小协作、基本口令与最小校验命令，完成复杂需求的最小稳定执行闭环。**
