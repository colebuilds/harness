# Codex PIR 工作流

## 1. 文档定位

本文档用于说明 Codex PIR 协作体系中的阶段流转方式。

本文档重点回答的问题是：

- 一个任务从开始到结束应如何流转
- 功能开发、Bug 修复、复杂任务的流程有何不同
- 什么时候应进入下一阶段
- 什么时候应回退上一阶段
- 阶段结束时应输出什么

本文档不负责说明：

- 三个角色的职责细节
- 具体 Prompt 文本
- `config.toml` 配置方式

这些内容分别由 `01-roles.md`、`04-prompts.md`、`03-configuration.md` 负责。

---

## 2. 工作流设计原则

PIR 工作流遵循以下原则：

### 2.1 先分析，后实现

任何进入代码修改的任务，都应先经过 `Planner` 阶段。

### 2.2 先计划，后落地

在 `Implementer` 开始修改前，应已经具备清晰、可执行的计划。

### 2.3 先验证，再结束

实现完成不等于任务完成。  
任务在进入 `Reviewer` 并完成审查前，不应视为结束。

### 2.4 复杂度影响流程，不影响核心角色

无论是功能开发、Bug 修复还是复杂任务，核心角色仍然是：

- `Planner`
- `Implementer`
- `Reviewer`

复杂度变化应通过：

- `Planner` 的 orchestration mode
- `Implementer` 的 debug mode

来吸收，而不是增加常驻角色。

### 2.5 回退优于硬撑

如果在执行或审查中发现前提错误、边界变化、风险扩大，应及时回退到前一阶段，而不是在错误前提上继续推进。

---

## 3. 功能开发工作流

## 3.1 Trigger

以下场景通常进入功能开发工作流：

- 新功能开发
- 页面功能扩展
- 新模块接入
- 业务逻辑新增
- 文档明确、目标清晰的功能迭代

## 3.2 Stage 1: Planner

`Planner` 阶段需要完成：

- 理解需求和上下文
- 明确本次目标
- 明确本次不包含
- 划定影响范围
- 输出分步骤计划
- 说明验证方式

阶段产出至少应包括：

- 目标
- 非目标
- 影响范围
- 步骤计划
- 风险点
- 验证方案

## 3.3 Stage 2: Implementer

在 `Planner` 输出清晰计划后，进入 `Implementer`。

`Implementer` 阶段需要完成：

- 按计划实施代码修改
- 仅在必要范围内扩展实现
- 补充必要测试
- 运行最小必要验证

阶段产出至少应包括：

- 实际代码修改
- 新增或更新的测试
- 执行过的验证
- 若有偏差，对计划变化的说明

## 3.4 Stage 3: Reviewer

实现完成并完成基本验证后，进入 `Reviewer`。

`Reviewer` 阶段需要完成：

- 检查实现是否符合计划
- 检查是否存在 bug、回归或边界遗漏
- 检查测试是否不足
- 给出审查结论

阶段产出至少应包括：

- Findings
- 风险等级
- 缺失项
- 是否建议回退
- 是否可结束任务

## 3.5 Exit Conditions

功能开发任务在满足以下条件后可结束：

- 目标实现完成
- 关键验证已通过
- `Reviewer` 已完成审查
- 无必须立即处理的 blocker

## 4. Bug 修复工作流

## 4.1 Trigger

以下场景通常进入 Bug 修复工作流：

- 已知 bug 修复
- 用户反馈异常行为
- 回归问题排查
- 页面逻辑不符合预期
- 接口、状态、边界条件引发的异常

## 4.2 Stage 1: Planner

Bug 修复任务中，`Planner` 负责的重点与功能开发不同。

`Planner` 阶段应重点明确：

- 问题现象
- 复现条件
- 影响范围
- 修复目标
- 修复后如何验证

阶段产出至少应包括：

- Bug 描述
- 复现条件
- 影响面
- 预期修复结果
- 验证标准

## 4.3 Stage 2: Implementer in debug mode

Bug 修复阶段仍由 `Implementer` 负责，但应进入 debug mode。

该阶段推荐顺序如下：

1. 复现问题
2. 收集证据
3. 判断根因
4. 实施修复
5. 验证修复
6. 检查是否引入副作用

该阶段的核心要求是：

- 不在未复现前盲改
- 不把现象描述当根因
- 不在证据不足时扩大修改

阶段产出至少应包括：

- 复现结果
- 根因判断
- 修复方案
- 实际修改
- 修复验证结果
- 若有必要，回归测试补充

## 4.4 Stage 3: Reviewer

Bug 修复完成后，`Reviewer` 应重点检查：

- 修复是否真正覆盖原问题
- 是否仅修补表象而未解决根因
- 是否引入新的副作用
- 是否需要补更多测试
- 是否存在边界遗漏

阶段产出至少应包括：

- 修复有效性判断
- 副作用风险
- 回归风险
- 测试缺口
- 是否通过审查

## 4.5 Exit Conditions

Bug 修复任务在满足以下条件后可结束：

- 原问题已可验证地消失
- 修复原因与修复措施逻辑一致
- 必要回归检查已完成
- `Reviewer` 未发现阻塞性风险

## 5. 复杂任务工作流

## 5.1 Trigger

以下场景通常进入复杂任务工作流：

- 涉及多个模块
- 涉及跨目录修改
- 既有前端页面改动，也有状态、接口、测试联动
- 预计改动文件较多
- 存在可拆分的多个子问题
- 一次性直接实现风险过高

## 5.2 Stage 1: Planner in orchestration mode

复杂任务首先由 `Planner` 接管，并进入 orchestration mode。

此阶段的重点不是直接产出细节实现，而是先决定：

- 是否需要拆分
- 哪些部分可以并行
- 哪些部分必须串行
- 每个子任务的边界是什么
- 每个阶段的输出和依赖是什么
- 最终如何汇总和审查

阶段产出至少应包括：

- 总目标
- 子任务划分
- 并行/串行判断
- 各子任务输入输出
- 风险集中点
- 汇总与验证方式

## 5.3 Stage 2: Implementer

在 orchestration mode 完成拆分后，再由 `Implementer` 按分解后的任务执行。

该阶段可以表现为：

- 单次实现一个子任务
- 分阶段实现多个子任务
- 在每个阶段后返回统一检查点

核心要求：

- 不跨子任务边界“顺手做完”
- 每次实现都应和拆分计划对齐
- 若子任务边界失效，应回退 `Planner`

## 5.4 Stage 3: Reviewer

复杂任务的 `Reviewer` 不仅审代码，还要审：

- 最终实现是否符合整体计划
- 子任务整合后是否出现接口错位
- 是否出现跨模块副作用
- 验证是否只覆盖局部、未覆盖整体

该阶段建议同时关注：

- 子任务质量
- 最终整合质量
- 测试与验证闭环是否完整

## 5.5 Exit Conditions

复杂任务在满足以下条件后可结束：

- 计划中的关键子任务已完成
- 各阶段输出已整合
- 关键联调或验证已完成
- `Reviewer` 已完成整体审查

---

## 6. 阶段完成判定

为避免阶段边界失效，每个阶段应有明确完成判定。

### 6.1 Planner 完成判定

满足以下条件时可视为 `Planner` 完成：

- 目标清晰
- 非目标清晰
- 影响范围清晰
- 实施路径清晰
- 验证方式清晰
- 若为复杂任务，拆分与阶段顺序清晰

### 6.2 Implementer 完成判定

满足以下条件时可视为 `Implementer` 完成：

- 计划内修改已落地
- 基本验证已执行
- 改动边界未失控
- 已具备可审查性

### 6.3 Reviewer 完成判定

满足以下条件时可视为 `Reviewer` 完成：

- 关键问题已识别
- 是否回退已有明确结论
- 是否可结束任务已有明确结论

## 7. 回退与重规划规则

PIR 不要求任务线性到底。  
当发现前提变化或实现偏差时，应允许回退。

### 7.1 回退到 Planner 的场景

以下情况应回退到 `Planner`：

- 初始计划关键前提错误
- 影响范围明显扩大
- 原本简单任务实际变为复杂任务
- 子任务边界无法成立
- 用户目标发生变化

### 7.2 回退到 Implementer 的场景

以下情况应回退到 `Implementer`：

- `Reviewer` 发现 bug
- `Reviewer` 发现回归风险
- 测试覆盖不足
- 实现与计划不一致
- Bug 修复未真正解决根因

### 7.3 重规划原则

当回退到 `Planner` 时，不应只做局部补丁式调整。  
应重新确认以下内容：

- 是否需要重定义边界
- 是否需要重新排序步骤
- 是否需要拆分任务
- 是否需要修改验证方案

---

## 8. 交付前验证要求

无论任务类型如何，在进入最终结束前，都应至少检查以下内容：

- 是否完成计划内目标
- 是否有计划外改动
- 是否执行了与本次修改匹配的验证
- 是否存在未解释的风险
- 是否存在明显缺失的测试
- 是否已完成 `Reviewer` 审查

对于前端任务，通常还应优先检查：

- 页面行为是否符合预期
- 关键交互是否可用
- 移动端或响应式是否受影响
- 样式和状态切换是否出现异常
- 修改是否影响已有页面结构

---

## 9. 一句话总结

PIR 的工作流不是“一个 agent 从头干到尾”，而是让任务按阶段稳定流转：

- 功能开发：`Planner -> Implementer -> Reviewer`
- Bug 修复：`Planner -> Implementer(debug mode) -> Reviewer`
- 复杂任务：`Planner(orchestration mode) -> Implementer -> Reviewer`

阶段清晰，回退明确，才能让 Codex 在复杂任务中保持稳定。
