# 将想法打磨成设计

通过自然、协作式对话，把模糊想法转化为完整设计和规格说明。

先理解当前项目上下文，再一次只问一个问题来收敛需求。  
一旦你理解了要构建什么，就把设计展示给用户并取得批准。

<HARD-GATE>
在你展示设计并得到用户批准之前，不要调用任何实现型 skill，不要写代码，不要搭脚手架，也不要采取任何实现动作。无论任务看起来多简单，这条都成立。
</HARD-GATE>

## 反模式：“这个太简单了，不需要设计”

所有项目都必须经过这个流程。  
待办列表、小工具函数、配置调整都一样。真正浪费时间的，往往就是那些“看起来很简单”的任务里未经审视的假设。

设计可以很短，但必须存在，并且必须先得到批准。

## 清单

必须按顺序完成以下事项：

1. **探索项目上下文** - 查看文件、文档、最近提交
2. **提供可视化辅助**（如果后续问题会涉及视觉内容）- 必须单独一条消息发送
3. **提出澄清问题** - 一次一个，理解目标 / 约束 / 成功标准
4. **提出 2-3 种方案** - 给出权衡和推荐
5. **展示设计** - 按复杂度分段展示，并逐段获得确认
6. **写设计文档** - 保存到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` 并提交
7. **规格自检** - 检查占位符、矛盾、歧义、范围
8. **让用户审阅书面 spec**
9. **转入实现** - 调用 `writing-plans`

## 过程要求

### 理解需求

- 先查看当前项目状态（文件、文档、最近提交）
- 先判断范围：如果请求包含多个相互独立的子系统，要先指出这一点
- 如果项目太大，不能用一个 spec 承接，就先帮用户拆成子项目
- 对合适范围的需求，一次只问一个问题
- 尽量优先多选题，但开放问题也可以
- 关注：目的、约束、成功标准

### 探索方案

- 至少提出 2-3 个方案并说明取舍
- 先讲推荐方案，再解释理由

### 展示设计

- 在你认为已经理解需求后，开始展示设计
- 每段长度按复杂度控制：简单事项几句话，复杂事项可达 200-300 词
- 覆盖：架构、组件、数据流、错误处理、测试
- 每个部分展示后，都询问用户是否认可

### 为隔离性和清晰度而设计

- 把系统拆成职责清晰、接口边界明确的小单元
- 每个单元都应该能回答：
  - 它做什么
  - 如何使用它
  - 依赖什么
- 如果不看内部实现就不能理解单元作用，说明边界不清
- 小而明确的单元更利于推理，也更利于稳定修改

### 在现有代码库中工作

- 先理解现有结构，再提出改动建议
- 遵循已有模式
- 如果现有代码的结构问题会影响当前任务，可以把“与当前目标强相关”的改进纳入设计
- 不要做无关重构

## 设计之后

### 文档化

- 把验证后的设计写入：
  - `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
- 如果有用户偏好路径，以用户偏好为准
- 如可用，使用 `elements-of-style:writing-clearly-and-concisely`
- 将设计文档提交到 git

### 规格自检

检查以下 4 项，并直接修正文档：

1. **占位符扫描** - 是否有 `TBD`、`TODO`、空白段落、模糊要求？
2. **内部一致性** - 各章节是否冲突？架构与功能描述是否一致？
3. **范围检查** - 这个 spec 是否过大，是否该继续拆分？
4. **歧义检查** - 是否存在多种解释？如果有，选定一种并写明确

### 用户审阅关口

写完并自检后，告诉用户：

> “Spec 已写入并提交到 `<path>`。请先审阅，确认没有要改的地方，再开始写 implementation plan。”

如果用户要求修改，先改文档并重新做自检，只有得到批准后才可继续。

### 进入实现

- 下一步必须调用 `writing-plans`
- 不要跳到其他 skill

## 关键原则

- **一次一个问题**
- **尽量多选题**
- **狠抓 YAGNI**
- **始终比较多个方案**
- **渐进验证**
- **必要时回退再澄清**

## Visual Companion

如果接下来很可能出现 mockup、布局、图示等视觉内容，可以先单独发一条消息邀请用户开启视觉辅助：

> “Some of what we're working on might be easier to explain if I can show it to you in a web browser. I can put together mockups, diagrams, comparisons, and other visuals as we go. This feature is still new and can be token-intensive. Want to try it? (Requires opening a local URL)”

这条消息必须独立发送，不能夹带其他内容。  
如果用户接受，再去读：

- `skills/brainstorming/visual-companion.md`
