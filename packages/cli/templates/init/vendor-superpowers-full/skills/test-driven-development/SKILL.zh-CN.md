# 测试驱动开发（TDD）

## 概览

先写测试。看它失败。再写最小代码让它通过。

**核心原则：**  
如果你没有亲眼看到测试先失败，就无法确认它真的在测你想测的东西。

## 适用时机

**始终使用：**

- 新功能
- Bug 修复
- 重构
- 行为变更

**例外（必须问 human partner）：**

- 一次性原型
- 自动生成代码
- 纯配置文件

## 铁律

```text
没有失败测试，就不准写生产代码
```

如果你先写了代码：

- 删掉它
- 从测试重新开始

不要把那些代码当“参考”保留。

## Red-Green-Refactor

### RED

写一个最小失败测试，清晰描述一个行为。

要求：

- 只测一个行为
- 名字清楚
- 尽量用真实代码，不滥用 mock

### 验证 RED

必须运行测试并确认：

- 它确实失败
- 失败原因符合预期
- 失败是因为功能缺失，而不是拼写错误或测试本身坏掉

如果测试一开始就通过，说明你没测到新行为。

### GREEN

写**最小**实现让测试通过。

不要顺便：

- 加额外功能
- 做额外重构
- 设计未来扩展

### 验证 GREEN

再次运行测试并确认：

- 新测试通过
- 其他测试没有被打坏
- 没有奇怪警告或错误

### REFACTOR

只有在 GREEN 之后才允许重构：

- 去重
- 改名
- 抽 helper

并且要保持测试全绿。

## 好测试的标准

- **最小化**：一个测试只讲一件事
- **清晰**：名称直接描述行为
- **表达意图**：测试应展示期望 API 和期望行为

## 常见借口

| 借口                       | 现实                                 |
| -------------------------- | ------------------------------------ |
| “太简单了，不用测”         | 简单代码一样会坏。                   |
| “我先写完再补测试”         | 那不是 TDD。                         |
| “我手动测过了”             | 手测不可重复，也不系统。             |
| “删掉先写的代码太浪费”     | 保留无法信任的代码才是真的浪费。     |
| “先写后测也能达到一样效果” | 不是。测试先行和测试补写不是一回事。 |

## 底线

不要把“写了测试”与“TDD”混为一谈。  
只有“先失败，再实现，再验证”才叫 TDD。
