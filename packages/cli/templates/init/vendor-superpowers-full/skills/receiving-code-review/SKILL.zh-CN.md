# 处理代码评审反馈

收到 review 不是流程结束，而是改进实现的一部分。  
你的目标不是“解释自己为什么这么写”，而是准确判断哪些问题成立，并把流程闭环。

## 核心原则

- 先判断反馈是否成立
- 成立的问题按优先级修复
- 修完后重新验证
- 必要时请求复审

## 处理顺序

1. 通读所有反馈，不要先急着反驳
2. 按严重程度分类：
   - `Critical`
   - `Important`
   - `Suggestion`
3. 对每一条反馈做判断：
   - 成立
   - 部分成立
   - 不成立
4. 对成立的反馈逐项修复
5. 运行验证
6. 汇总处理结果并准备复审

## 如何回应

### 反馈成立

- 直接修复
- 不要解释太多
- 用结果闭环

### 反馈部分成立

- 说明你接受的部分
- 说明未接受的部分及原因
- 尽量附证据

### 反馈不成立

- 用技术理由说明
- 引用代码、测试、计划或需求作为证据
- 保持简洁，不要情绪化

## 红旗

- 把 review 当成辩论赛
- 试图通过“解释意图”替代修复问题
- 修完后不重新验证
- 有改动但不请求复审
- 把多个反馈糊成一个模糊结论

## 最终目标

review 的完成条件不是“你回复了”，而是：

- 问题被修复或被技术性驳回
- 验证已重新运行
- reviewer 或流程已确认可以继续
