# 完成前验证

## 概览

没有验证就宣称完成，不是高效，而是不诚实。

**核心原则：**  
先有证据，再做结论。

## 铁律

```text
没有新鲜的验证证据，就不准声称完成
```

如果你没在这条消息所对应的工作中实际运行验证命令，就不能说“通过了”。

## 闸门函数

在任何成功表述前，必须做：

1. **识别**：什么命令能证明这条结论？
2. **执行**：运行完整命令
3. **阅读**：查看完整输出和退出码
4. **确认**：输出是否支持你的结论？
5. **然后才允许**说“完成 / 通过 / 修好”

跳过任何一步都不算验证。

## 常见场景

| 结论         | 需要什么证据             | 什么不算            |
| ------------ | ------------------------ | ------------------- |
| 测试通过     | 测试命令输出 0 失败      | 上一次运行结果      |
| Lint 通过    | linter 输出 0 错误       | 只跑了部分检查      |
| 构建成功     | build 命令 exit 0        | “看起来应该行”      |
| bug 修好了   | 原始问题复现路径现在通过 | 只改了代码          |
| agent 完成了 | diff + 验证              | 只听 agent 口头汇报 |

## 红旗

- 使用 “应该”“大概”“看起来”
- 在验证前表达满意或成功
- 提交 / PR 前没验证
- 只相信 agent 的“已完成”
- 只做部分验证

## 典型模式

### 测试

✅ 运行测试，看见全通过，再说“测试通过”  
❌ “现在应该过了”

### 回归验证

✅ 写测试 -> 看到失败 -> 修复 -> 再看到通过  
❌ “我已经写了回归测试”

### 构建

✅ 跑构建并看到 exit 0  
❌ 只看 lint 通过

### 需求完成

✅ 回读计划，逐项勾对  
❌ “测试都过了，所以阶段完成”

## 为什么这么重要

很多失败都源于：

- 过早说“好了”
- shipping 了未定义函数
- 需求没做全却说完成
- 误把“改了代码”当“解决了问题”

## 适用时机

在以下所有时刻都应触发：

- 任何“完成 / 成功 / 修复”相关表述前
- commit 前
- push / PR 前
- 任务切换前
- 进入下一阶段前

## 底线

运行命令。看输出。再下结论。
