# 系统化调试

## 概览

随机尝试修法会浪费时间，也会制造新 bug。  
快速补丁通常只掩盖症状，而不解决根因。

**核心原则：**  
在完成根因调查前，不要提出修复方案。

## 铁律

```text
没有根因调查，就不准修
```

## 适用范围

用于任何技术问题：

- 测试失败
- 线上 bug
- 异常行为
- 性能问题
- 构建失败
- 集成问题

尤其在以下场景更不能跳过：

- 时间压力很大
- 你觉得“先试个 quick fix”
- 已经试了多个修法
- 上一个修法没生效
- 你并不真正理解问题

## 四个阶段

### 第一阶段：根因调查

在任何修复动作前，必须先做：

1. **仔细读错误信息**
   - 不要跳过 warning 或 stack trace
   - 记录文件、行号、错误码

2. **稳定复现**
   - 能稳定触发吗？
   - 精确步骤是什么？
   - 如果不能稳定复现，先继续收集数据

3. **检查最近变更**
   - 最近改了什么？
   - 查看 git diff、最近提交、配置变更、依赖变更

4. **多组件系统先加证据采样**
   - 在边界处记录输入输出
   - 验证配置是否穿透
   - 查状态在哪一层出错

5. **追踪数据流**
   - 异常值从哪来？
   - 谁传进来的？
   - 一直向上追，直到找到源头

### 第二阶段：模式分析

在修之前先比较：

1. 找同代码库中正常工作的类似实现
2. 完整读参考实现
3. 列出差异
4. 识别依赖和隐含假设

### 第三阶段：假设与验证

1. 形成**单一假设**
   - “我认为根因是 X，因为 Y”
2. 做最小改动验证这个假设
3. 验证结果：
   - 成功 -> 进入实现阶段
   - 失败 -> 回到调查，形成新假设
4. 如果你其实不懂，就明确说不懂，继续查，不要装懂

### 第四阶段：实施修复

1. 先写失败测试或最小复现
2. 只修一个根因
3. 验证修复
4. 如果失败：
   - 少于 3 次，回到第一阶段
   - 3 次及以上，开始怀疑架构

### 第四点五阶段：3 次失败后质疑架构

如果你已经试了 3 次以上修法，且每次都暴露出新的共享状态、耦合或别处问题：

- 不要继续盲修
- 要开始问：
  - 这套结构本身是不是错了？
  - 我们是不是在用惯性维持一个坏模式？

此时应与 human partner 讨论，而不是继续堆补丁。

## 红旗

出现下面这些想法时，立刻回到第一阶段：

- “先 quick fix，回头再查”
- “我改几个地方试试”
- “先不写测试，手动看看”
- “大概就是 X，我直接修”
- “我不完全懂，但应该这样”
- “再试一次修法”

## 用户提示你做错时的信号

如果 human partner 说：

- “那真的发生了吗？”
- “能不能证明给我看？”
- “别猜”
- “再深想一层”

说明你在跳过系统化调试。

## 底线

调试不是“找一个看起来能过的修法”，而是找到真正根因，并在证据支持下修复。
